    Why it matters. The agent works faster and more reliably when it searches and runs things for real, rather than reasoning about the code from memory.

    Methodology & honesty

    How every number was computed (click to expand)

    Only real prompts are scored. A “user” record counts as a prompt only if it is not a tool-result, not a subagent (sidechain) turn, not meta/injected, not a slash-command stub, and not a paste/system-prompt over 6,000 chars or opening with “You are …”. This removes the contamination that made the old tool report a 106-vs-real average.

    Everything is a rate, then squashed. Each dimension is a per-prompt or per-opportunity rate run through min(1, rate/target), so doing more work never raises the score — only doing it better does. Weights: Briefing 24%, Verification 22%, Context-setting 22%, Iteration 18%, Toolcraft 14%.

    Thin signals are hedged, not faked. Each dimension is pulled toward a neutral 50 in proportion to how many opportunities it had (e.g. Iteration had only 4 corrections, so it is flagged “low data”). Both raw and confidence-adjusted scores are shown.

    Archetype describes your driving style, not the collaboration's quality, so it is built on a separate agency-weighted vector: Briefing, Iteration, Toolcraft and Delegation (handoffs to subagents/background jobs/planning) count fully, while Verification and Context — habits Claude largely does on its own — are discounted (35% and 15% weight). It is the nearest prototype by cosine on z-scored values; if the top two are within 0.06 we show a blend. Active time caps idle gaps at 5 min. Fixes vs v1: prompt mis-count, length inflation, idle-time over-count, random archetype, uncapped tool-diversity, and keyword “error” false-positives.

    Limits: this measures observable behavior, not intent; detectors are heuristic and English-biased; it's a single snapshot, not a trend. Terse prompts that carry intent from the prior turn can under-score Direction.
